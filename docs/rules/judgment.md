# 判斷力外化（D）

讀者：日常 session 的 Sonnet。這裡是高階模型預先做好的判斷，照觸發條件執行。

## 1. 何時算「真的完成」

觸發：你準備回報「完成」之前。
判準（全部通過才算完成）：
- [ ] `npx vite build` 通過（改了 src/ 或 api/ 才需要）
- [ ] 任務的每一點要求都有對應改動，或明說「查過、此點不需改」
- [ ] CLAUDE.md 的「同步配對表」逐條核對過，回報中寫出核對結果
- [ ] `git diff` 自查過：沒有與任務無關的改動混入

正例：「已改 BASE_IP_POST；配對表核對：定位條有改、checkItems 有改、IP_TYPES 副標
不需改（新規則沒動類型定義）；build 通過。」
反例：「build 通過了，完成！」——build 只驗語法，篩選邏輯錯了 build 照樣過。

## 2. 何時停下來問使用者

使用者已授權日常改動（含 push main）都直接做。只有命中下列才停：
- 需求有兩種以上合理解讀，且做錯的重做成本高（整頁重寫等級）
- 發現使用者的要求與現有資料矛盾（例：要刪的東西正被別處依賴）
- 要刪除或覆蓋「不是這次任務建立」的內容，且刪了難復原

正例：使用者說「把廢文風拿掉」，但收藏區 8 成是廢文風貼文 → 問一次再動手。
反例：按鈕顏色沒指定色號 → 沿用同分頁既有色系，自己決定，不問。

## 3. 什麼訊號代表方向錯了、該換路

觸發：任一命中就停止目前做法。
- 同一個錯誤修第二次仍未好 → 停，重新診斷根因，不做第三次同款嘗試
- diff 規模超過任務描述的合理範圍（加個按鈕卻改了 200 行）→ 回退，重想切入點
- 開始猜測而不是查證（「應該是這樣」出現兩次）→ 回去讀碼或問使用者

正例：複製功能修了兩次還是壞 → 停下寫出根因假設清單，逐一驗證，而不是再試一版。
反例：build 錯一次就宣告方向錯 → 讀錯誤訊息修掉即可，這不是換路訊號。

## 4. 何時建議升級模型

見 dispatch.md「升降級路徑」。本檔不重複，以該檔為準。

## 5. 品質底線怎麼驗（按改動類型）

**改 UI／功能**：build 必過 → 逐條自查任務檢查點 → 樣式沿用同分頁既有模式
（色系、圓角、字級照抄鄰近元件，優先複製既有 style 而不是自創）。
**改 prompt 常數**：新規則與 TONE_RULE、該 prompt 既有【禁止事項】逐條對照，
有打架先回報矛盾點再改；字數與格式規則前後一致（不能上面說 100 字下面例子 300 字）。
**上線**：push main 即上線（Vercel 自動部署），push 前 build 必過。
**使用者貼長框架文件要求升級 prompt**：先列「框架章節 → 落點」對照清單再動手，
每一節標「已納入／合併進X／未納入＋原因」，防漏節。

## 6. 誠實條款（原樣執行）

- 測試失敗就說失敗並貼輸出；跳過的步驟明說跳過
- 查不到的事標「未確認」，優先寫「未確認」而不是編一個合理答案
- 品味判斷（文案好不好、語氣像不像）你補不了，交還使用者判斷，
  你只回報「符合／不符合既有規則哪幾條」
