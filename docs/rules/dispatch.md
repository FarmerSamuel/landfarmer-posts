# 模型調度守則（C）

讀者：日常 session 的 Sonnet。觸發時機：你考慮派 subagent 時，或任務超出你的等級時。

## 本專案何時需要派工（先看這條再往下讀）

本專案是單檔小應用，多數任務（改一段 prompt、加一個按鈕）**直接自己做，不派工**。
只有命中下列情況才派：
1. 高風險改動的獨立驗證（見「驗證不自驗」）。
2. 需要掃描超過 10 個檔案或超過 3,000 行的搜尋、盤點（本專案罕見）。
3. 使用者明確要求多方案比較或審查。

正例：使用者要求「重構 App.jsx 拆成多檔」→ 改完派一個驗證 agent 核對功能無回歸。
反例：加一個篩選按鈕也派 agent 去「先研究一下」→ 純開銷，直接做。

## 派工三要素（缺一不發）

每次派工的 prompt 必須包含：
1. **目標與動機**：做什麼、為什麼做（讓 agent 能自行處理小歧義）。
2. **驗收條件**：怎樣算完成，可檢查（如「build 通過」「列出檔案:行號」）。
3. **回報格式**：規定只回結論與證據，禁止整檔傾倒（見回報合約）。

## model 內聯（每次派工顯式指定，不寫就等於沒分級）

| 等級 | model 值 | 用途 |
|------|----------|------|
| 低階 | `haiku` | 機械掃描：找字串、核對檔案存在、跑指令收集輸出 |
| 中階 | `sonnet` | 一般實作、盤點、驗證、評分 |
| 高階 | `opus` 或 `fable` | 對抗審查、架構評估（額度貴，見升降級路徑） |

本環境的 Agent 派工介面只有 `model` 欄位、**沒有 effort 欄位**，不要在派工參數裡
填 effort（會報錯）；推理力度需求寫進 prompt 文字（如「請深入檢查邊界情況」）。

## 回報合約（寫進每個派工 prompt 的固定尾巴）

> 回報只包含：(1) 結論一句話 (2) 證據清單，格式「檔案:行號 — 一句說明」
> (3) 你不確定的地方。禁止貼整個檔案內容，diff 只貼改動的部分。

## 驗證不自驗（鐵律）

寫程式的 agent 不能同時當自己的驗證者。命中下列任一情況，改完後派一個
**全新、不知道實作過程**的 sonnet agent 做驗證：
- 改動超過 100 行
- 動到兩個以上分頁的共用邏輯（如 TONE_RULE、favorites/history）
- 使用者說過這裡之前壞過

驗證 agent 的 prompt 給它「任務原文＋檢查點清單」，不給它「實作者的自述」。
正例：重構複製功能後，派驗證 agent 逐分頁確認四個複製按鈕行為。
反例：把實作 agent 的回報再丟給它自己「確認一下」→ 它會說沒問題。

## 升降級路徑

- **升級（建議使用者切 Opus/Fable）**，命中任一：
  同一個 bug 修兩次都沒好；要動整體架構（拆檔、換框架）；
  資安相關（API key、注入）；使用者連續兩次對同一產出不滿意。
  做法：停下，向使用者說明「這題建議切高階模型，原因是 X」，等指示。
- **降級（改派 haiku）**：純機械、驗收條件可完全寫死的子步驟。
- 模型切換切在計畫邊界（一個任務完整結束後），不在任務中途切換。
