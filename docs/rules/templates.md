# 任務交辦範本（E）

讀者：日常 session 的 Sonnet。派工時複製對應範本、填空、發出。
`{}` 是填空處。model 值照範本寫死，除非 dispatch.md 升降級路徑另有指示。

## 1. 搜尋（model: haiku）

> 在 /home/user/landfarmer-posts 用 Grep/Glob 找出 {要找的東西}。
> 動機：{為什麼要找}。
> 回報格式：每筆「檔案:行號 — 該行內容一句摘要」，找不到就回「查無，已試過
> 關鍵字：{列出}」。禁止貼大段檔案內容，禁止順手修改任何檔案。

## 2. 實作（model: sonnet）

> 你在 /home/user/landfarmer-posts 專案工作。先讀 CLAUDE.md 並遵守其中的
> 檔案地圖與同步配對表。
> 任務：{使用者原話或明確需求}。
> 驗收條件：npx vite build 通過；{任務專屬檢查點}；同步配對表核對完成。
> 回報格式：(1) 改了哪些檔案:行號 (2) 配對表核對結果逐條列出 (3) build 輸出
> 最後三行 (4) 不確定的地方。不要 commit、不要 push（由主 session 決定）。

## 3. 重構（model: sonnet；超過 300 行的重構先按 dispatch.md 評估是否建議升級）

> 你在 /home/user/landfarmer-posts 專案工作。先讀 CLAUDE.md。
> 重構目標：{目標}。行為不可改變：重構前後功能完全一致。
> 驗收條件：npx vite build 通過；逐一列出「原功能 → 新位置」對照表；
> git diff 中不得混入行為改變。
> 回報格式：對照表＋build 結果＋你認為有風險的點。不要 commit、不要 push。

## 4. 研究（model: sonnet；純網路查證可降 haiku）

> 研究問題：{問題}。動機：{為什麼需要知道}。
> 規則：查證不憑記憶，給出處；查不到就標「未確認」，禁止編造。
> 回報格式：(1) 結論三句以內 (2) 依據清單（出處/檔案:行號） (3) 未確認事項。

## 5. 審查（model: sonnet；上線前高風險審查用 opus）

> 審查對象：{diff 或檔案}。你是獨立審查者，沒有參與實作，請試著找出問題
> 而不是確認沒問題。
> 檢查：(1) 任務檢查點 {貼上} 逐條核對 (2) CLAUDE.md 同步配對表有無漏改
> (3) 有無與任務無關的改動混入 (4) prompt 改動與 TONE_RULE 有無矛盾。
> 回報格式：每個問題「檔案:行號 — 問題 — 建議修法」；沒問題的項目也要列出
> 「已核對：X」。禁止只回「看起來沒問題」。

## 6. 交接 HANDOFF（context 快滿或換模型前，寫進回報或存檔）

> ## HANDOFF
> - 目標：{這個任務最終要達成什麼}
> - 當前狀態：{做到哪，哪些已驗證}
> - 已變更檔案：{檔案:行號清單，含未 commit 的}
> - 決策紀錄：{做過的關鍵選擇與原因，一行一條}
> - 失敗事項：{試過但不行的路，防止重蹈}
> - 精確下一步：{下一個動作，具體到可直接執行}

觸發時機：context 使用超過七成、或使用者說要換模型時，主動輸出 HANDOFF。
