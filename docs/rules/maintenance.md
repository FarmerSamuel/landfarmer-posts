# 維護協議（F）

讀者：日常 session 的 Sonnet。觸發時機：要改制度檔、踩了坑、或模型大版本更新後。

## 哪些檔可以自行改

- **隨時可寫**：docs/quality/ 底下的紀錄類檔案（golden-tasks.md 的紀錄表、
  ledger.md、diagnosis.md 的補充觀察）——這些是帳本，照格式追加即可。
- **改前要在回報中說明理由**：CLAUDE.md、docs/rules/*.md——這些是規則正本。
  修改必須符合：(1) 一併說明「為什麼現有規則不夠」 (2) 通過 scripts/check-rules.sh
  (3) 新規則先過 Outgrowth 檢查：「Sonnet 沒有這條也會做對嗎？」會，就不加。
- **不可自行刪規則**：刪除或淘汰規則只能列提案給使用者，批准後才動。

## 踩坑教訓寫回哪裡

出現「使用者糾正了你一次」的情況，當下就做：
1. 判斷坑的類型：配對漏改 → 補進 CLAUDE.md 同步配對表；
   判斷失誤 → judgment.md 對應節加一組正例/反例；派工失敗 → dispatch.md。
2. 寫不進上述任何一處的，記到 ledger.md 觀察區。
3. 同一個坑第二次出現＝規則寫得不夠明確，重寫該條並讓表述更可檢查。

## 負向結果帳本（docs/quality/ledger.md）

記「經測試證實沒效或純開銷」的規則。格式：日期／規則位置／證據／建議處置。
同一檔累積 3 條 → 向使用者提出刪除提案。規則是負資產：佔 context、僵化強模型。

## 審計模式：什麼時候整套重審

觸發任一：(1) Claude 模型大版本更新（如 Claude 6 出現）
(2) 黃金任務重跑成績比基準線退步 (3) 使用者覺得「AI 最近變笨了」。
做法：用當時可用的最強模型開一個 session，指令：「審計本專案全部制度檔：
找矛盾、過時、為舊模型缺陷寫的多餘護欄、純開銷候選，每條附檔案:行號＋建議處置。
你找問題，使用者決定刪改。」

## 回聘協議：什麼題目值得回頭用高階模型（Opus/Fable，額度貴）

值得：深度審查（資安、整體架構評估、prompt 系統的全面矛盾檢查）、
制度審計、連續失敗後的根因診斷。做法：高階模型只當診斷者/審查者，
產出可執行的分步計畫；執行拆單給 Sonnet。切換點在計畫邊界，不在任務中途。
高階審查 session 盡量唯讀（不給它改檔，只給它報告義務），界線在派工設定裡擋。

不值得：日常改功能、調 prompt、修一般 bug——Sonnet 實測基準已證明能做好。

## 盲測重跑時機

制度檔大改（新增/重寫整個章節）後，抽 1-2 個黃金任務按 golden-tasks.md
的盲測流程重跑，數據記入紀錄表。持平且 token 下降＝通過；退步＝回退改動。
