# 給未來 session 的信（G）

寫於 2026-07-04，Fable 5 制度建立 session。讀者：未來任何接手本專案的模型與使用者。

## 三件使用者沒問、但最重要的事

1. **這個專案最大的風險不是程式，是 prompt 品質的靜默劣化。**
   八個 prompt 互相引用 TONE_RULE，規則一多就會互相打架（字數上限 vs 模組數量、
   禁止事項 vs 範例語氣）。每次改 prompt 都跑「矛盾對照」那條驗證，比 build 重要。
2. **使用者的內容框架會持續演進（已經歷 v1→v3.0）。**
   每次大改版都會留下「UI 還是舊版說法」的殘渣——IP_TYPES 比例副標就是現存實例。
   接到框架升級任務，第一動作是列章節對照清單，最後一動作是掃配對表。
3. **基準數據證明 Sonnet 單點任務全滿分（見 golden-tasks.md）。**
   所以不要為了「幫助弱模型」而加手把手教學規則——那是純開銷。
   制度只該管三件事：省 token（檔案地圖）、防漏改（配對表）、防假完成（驗證清單）。

## 制度最可能的退化方式

- CLAUDE.md 被逐次「順手加一條」撐爆 → 行數 hook 會擋，但擋不住 150 行內的
  低價值堆積；每季看一次 ledger.md 觀察區。
- 檔案地圖錨點與程式碼漂移（改名 BASE_* 常數後沒更新地圖）→ 改名時把
  CLAUDE.md 當成配對表的一項。
- 配對表變成儀式：回報寫了「查過」但其實沒查 → 抽查方式是看回報有沒有寫出
  「不需改的理由」，只寫「查過不需改」五個字的就是沒查。

## 信心最低的產出與原因

- **dispatch.md 的派工時機**：本專案太小，派工守則幾乎用不到；寫它主要為了
  未來專案變大。可能整檔都是 Outgrowth，留給審計模式裁決。
- **GT4（策略判斷型任務）沒有基準**：品味類任務 rubric 很難寫死，
  目前的檢查點偏形式（有沒有引用原文），測不出建議好不好。
- **Stop hook 在每個回合尾都跑**：目前腳本 <100ms 沒感覺，若未來變慢要改成
  只在 docs/ 或 CLAUDE.md 有變更時才檢查。
