# 環境診斷（A）— 2026-07-04，Fable 5 制度建立 session

根據本專案實際 session 歷史診斷，前三名問題與修法。修法已寫入 CLAUDE.md 成為規則。

## 第 1 名：最漏 token — 整檔重讀 1,100 行的 App.jsx

**現象**：全部邏輯與 8 個 AI prompt 都在單一千行級的 src/App.jsx。歷次 session 不論改多小，
都整檔 Read（約 3 萬 token），context 壓縮後又重讀一次。這是本專案最大的 token 漏洞。

**修法（弱模型照做）**：CLAUDE.md 內建「檔案地圖」——列出每個功能區塊的搜尋錨點
（例：改 IP 貼文規則 → Grep `BASE_IP_POST`，只讀該區段前後）。
規則寫死：先 Grep 定位，只讀需要的區段；單次 Read 超過 300 行前先問自己是否真的需要。

## 第 2 名：最易出錯 — prompt 與 UI 的隱性配對，改一邊漏一邊

**現象**：prompt 常數（如 BASE_IP_POST）和頁面文案（定位條、品質核對清單、主題卡副標）
內容互相呼應但分散在檔案不同處。實例：IP v3.0 升級需同步改 3 處；
弱模型高機率只改 prompt、忘了 UI，使用者要開口糾正才會補。

**修法（弱模型照做）**：CLAUDE.md 列「同步配對表」——改 A 必查 B 的寫死清單。
改任何 BASE_* 常數後，逐條核對配對表並在回報中說明「查過、有改／查過、不需改」。

## 第 3 名：最易假完成 — 驗證手段只有 build，語法過 ≠ 功能對

**現象**：`npx vite build` 只驗語法，抓不到篩選邏輯錯、prompt 規則互相矛盾。
而部署 = push main 直接上線（Vercel），錯了就是線上錯，沒有 staging。

**修法（弱模型照做）**：寫死分類驗證清單——
改 UI：build 必過＋逐條自查任務檢查點；
改 prompt：自查新規則與 TONE_RULE 及既有【禁止事項】有無打架，有打架先回報再改；
push main 前：build 必過（hook 腳本強制）。
