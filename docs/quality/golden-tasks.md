# 黃金任務與驗收基準（H）

本檔是制度成效的量尺。制度改版後，重跑黃金任務並與基準線對比，數據不進步就回退制度。

## 「一次糾正」的定義（寫死）

使用者必須開口指出、模型才會去修的缺陷，算一次糾正。
- 同一個原因造成的多處問題，算一次（例：忘了處理 fallback，四個複製按鈕都沒改 = 1 次）。
- build 失敗、任務要求沒達成、動到不該動的地方、樣式明顯不一致，都算。
- 使用者主動追加的新需求，不算糾正。

## 五個黃金任務（prompt 原文固定，不得改寫）

### GT1 調整 prompt 規則
> 把 FB文案分頁的『農夫版』hashtag 規則改成：固定 3 個 hashtag，且第一個必須是 #大地農夫。其他版本的規則不要動。

檢查點：(1) BASE_FARMER 的 hashtag 規則正確改寫 (2) 農婦版／散文版一字未動 (3) build 通過 (4) 沒有無關改動。

### GT2 新增網頁功能
> 在歷史頁（📋 歷史）加一排聲線篩選按鈕：全部／農夫版／農婦版／農婦散文風版。點了只顯示該聲線的紀錄，筆數顯示也要跟著變，按鈕樣式要跟系統現有按鈕一致。

檢查點：(1) 四個篩選選項齊全且邏輯正確 (2) 筆數跟著篩選變 (3) 樣式沿用綠色系既有按鈕模式 (4) build 通過 (5) 收藏按鈕等既有功能無回歸。

### GT3 修錯誤
> 有使用者反映在某些環境下，網站上所有『複製』按鈕按了都沒反應（提示：navigator.clipboard 在非 https 環境不可用）。幫全站所有複製功能加上可靠的備援方案，並確保複製真的失敗時使用者看得出來。

檢查點：(1) 全部四條複製路徑都覆蓋（copy／copyVideoScript／copyGentleScript／copyIPPost），優先抽共用函式 (2) fallback 技術正確（textarea + execCommand） (3) 失敗時有使用者可見回饋 (4) build 通過。

### GT4 內容策略（判斷型）
> 用三個真實農場情境，測試 IP 貼文 v3.0 的生成規則會產出什麼問題，提出最多 3 條具體修正建議。

檢查點：(1) 建議引用規則原文（可定位到哪一條） (2) 建議可直接執行、不是泛泛而談 (3) 沒有為改而改（若規則沒問題要敢說沒問題）。

### GT5 配對同步（專測診斷第2名的失敗模式）
> IP貼文的內容濃度『濃』改一下：字數上限從 300 字降到 250 字。

檢查點：(1) BASE_IP_POST 內【密度對照】的「濃」改為 200-250 字 (2) IP_DENSITIES 的
strong 選項 desc「200-300字」同步改為「200-250字」（UI 按鈕上看得到的字）
(3) build 通過。漏改任一處＝1 次糾正。

## 評分 rubric（寫死，評分者照填）

| 維度 | 配分 | 標準 |
|------|------|------|
| 正確性 | 0–4 | 任務要求每一點都達成＝4；每漏一點扣 1 |
| 完整性 | 0–3 | 相關聯動都處理（共用函式、提示文字、無回歸）＝3 |
| 需人工介入次數 | 0–3 | 0 次＝3、1 次＝2、2 次＝1、3 次以上＝0 |

勝負門檻：兩版總分差 ≥2 為勝；差 ≤1 為平。

## 盲測流程（寫死）

1. 同一黃金任務跑兩版：A＝無制度檔、B＝有制度檔（各自隔離 worktree，Sonnet）。
   A 版 worktree 要先刪掉 CLAUDE.md 與 docs/（check-rules.sh 已容忍缺檔，不會鎖死）。
2. 評分者是另一個不知情的 Sonnet agent：只給它「任務原文＋兩份匿名成果（隨機標為甲/乙）＋本 rubric」。
3. 評分者不得被告知哪版有制度、誰先誰後。
4. 分數與糾正次數記入下方紀錄表。
